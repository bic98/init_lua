## 목적

하루 동안 발생한 오류 해결 로그(`docs/error-logs/YYYY-MM-DD.md`)를 읽고, 각 오류에서 배울 수 있는 CS/웹 개발 개념을 CMU Professor 수준의 교육 문서로 변환한다. 퇴근 전에 호출하면 퇴근길/집에서 1시간 동안 읽으며 학습할 수 있는 자료가 생성된다.

## 인자 처리

- 인자가 없으면: 오늘 날짜의 로그 파일을 사용한다
- 날짜가 주어지면: 해당 날짜의 로그 파일을 사용한다 (예: `2026-04-08`)

## 실행 절차

### Step 1: 오류 로그 읽기

1. `nextjs-app/docs/error-logs/YYYY-MM-DD.md` 파일을 읽는다
2. 파일이 없으면 사용자에게 "오늘은 기록된 오류가 없습니다"라고 알린다
3. 각 오류 항목을 파싱하여 다음을 추출한다:
   - 오류 제목
   - 증상
   - 원인
   - 해결 방법
   - 관련 CS 개념 키워드

### Step 2: 학습 구조 설계

오류 개수와 관련 개념에 따라 학습 자료의 구조를 설계한다.

**원칙:**
- 오류 1-2개: 단일 문서 (1시간)
- 오류 3-5개: 2-3개 섹션으로 나눈 단일 문서 (1시간)
- 오류 6개 이상: 관련 개념별로 그룹핑하여 구조화 (1시간)

### Step 3: 학습 자료 생성

출력 파일: `nextjs-app/docs/error-logs/STUDY_YYYY-MM-DD.md`

## 작성 원칙

### 서술체
- "-한다", "-이다", "-된다" 체를 사용한다 (CMU Professor 스킬과 동일).
- 불릿 포인트 나열이 아니라 교과서 문단 형태로 풀어서 쓴다.

### 설명 깊이
- **3단계 설명**: 일상 비유 → 정식 정의 → 깊은 이해
- 핵심 CS 개념 하나당 최소 500자 이상의 서술형 설명
- 수식/다이어그램이 필요한 경우 반드시 포함

### 개념 분석 구조 (각 개념마다 필수 포함)
1. **상위 개념**: 이 개념이 속하는 더 넓은 범주
2. **하위 개념/응용**: 파생된 실무적 응용
3. **유사 개념과의 차이**: 비슷하지만 다른 것들과의 비교
4. **장단점**: 왜 장점인지, 왜 단점인지를 이론적으로
5. **실전 선택 가이드**: 언제 쓰고, 언제 피해야 하는지

### 연결과 맥락
- 오늘의 오류가 서로 어떻게 연결되는지를 찾아서 서술한다
- 어제/이전의 오류 로그가 있으면 참고하여 학습 연속성을 만든다
- "오늘 배운 X는 어제 배운 Y의 연장선에 있다" 형태의 연결

### 실전성
- 모든 설명에 실제 프로젝트 코드를 사용한다 (추상적 예시 X)
- `file_path:line_number` 형식으로 실제 코드 위치를 참조한다
- 오류의 실제 에러 메시지, 스크린샷 설명 등을 포함한다

### 분량 목표
- **전체**: 약 1시간 읽기 분량 (~8000-12000 단어)
- **오류당**: 약 15-20분 (~2500-4000 단어)
- 오류가 1개뿐이면 관련 개념을 더 깊이 파서 1시간을 채운다

### Mermaid 다이어그램
- 각 오류마다 최소 1개의 다이어그램 포함
- 오류 발생 흐름 (sequenceDiagram), 개념 관계 (flowchart), 비교 (classDiagram) 등
